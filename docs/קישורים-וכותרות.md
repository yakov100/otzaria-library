# קישורים וכותרות בספריית אוצריא — מבנה ויצירה

תיעוד זה מסביר איך נוצרים ארבעת סוגי ה"קישורים" שמופיעים ב־`seforim.db` של אוצריא, ומה בדיוק כל אחד מהם:

1. **קישורים רגילים** — קישור שורה↔שורה (מפרש, תרגום, מקור וכו').
2. **קישורים למילה** — קישור מעוגן לתו/מילה מסוימים בתוך שורה (כמו סמני האותיות ב"שער הציון").
3. **קישורי טווח** — קישור שמכסה כמה שורות (טווח פסוקים/סעיפים).
4. **קישורי "כותרות" / עליות** — מבנה כותרות חלופי כמו ה"עליות" תחת הפרשות בבראשית.

> **הבהרה חשובה לפני הכול:** רק שלושת הראשונים הם *קישורים* אמיתיים (רשומות בטבלת `link`). הרביעי — ה"עליות" — **אינו קישור בכלל**, אלא **מבנה תוכן חלופי** (alternate table of contents). הוא יושב בטבלאות נפרדות לגמרי (`alt_toc_*`). ההבחנה הזו קריטית להבנת השאר.

הקוד שמייצר את כל אלה חי ב־repo האחות **[SeforimLibrary](https://github.com/Otzaria/SeforimLibrary)** (Kotlin), לא ב־repo הזה. ה־repo הזה (`otzaria-library`) מספק את **הקלט**: קובצי הספרים והקישורים. כל הפניה לקוד למטה היא לנתיב בתוך `SeforimLibrary`.

## 0. מפה מהירה

| מה שרואים | טבלאות ב־DB | נוצר ע"י Sefaria? | נוצר ע"י אוצריא? |
|---|---|---|---|
| **קישורים רגילים** | `link` (+ `connection_type`) | ✅ | ✅ |
| **קישורים למילה** | `link` + `link_anchor` | ✅ (שני מסלולים) | ✅ (מסלול אחד, צד־מקור בלבד) |
| **קישורי טווח** | `link` + `link_range` + `link_coverage` | ✅ | ✅ (`line_index_X_end` ב־`_links.json`) |
| **כותרות / עליות** | `alt_toc_structure` + `alt_toc_entry` + `line_alt_toc` | ✅ | ✅ (`alt_toc/<ספר>_alt_toc.json`) |
| *(TOC רגיל — לשם השוואה)* | `tocEntry` + `tocText` + `line_toc` | ✅ | ✅ (מתגי `<h1>`–`<h6>`) |

מספרים אמיתיים מה־DB הבנוי (בניית Sefaria בלבד, ~4.3GB): 3,882,977 קישורים · 165,403 עוגני־מילה · 111,948 טווחים · 5,190,913 שורות כיסוי · 387 מבני־עליות · 61,339 רשומות־עליות.

**תשובה מהירה לשאלה "מה תומך הגנרטור של אוצריא":** כל ארבעת המנגנונים ✅ — קישורים רגילים, קישורים למילה (כמו ב"שער הציון"), קישורי טווח ועליות. פירוט מלא ב[פרק 7](#7-מטריצת-התמיכה) והוראות יצירה ב[פרק 8](#8-איך-ליצור-בפועל).

<a name="1-רקע"></a>
## 1. רקע: שני הגנרטורים ושרשרת הבנייה

`seforim.db` (הבסיס שאפליקציית אוצריא צורכת) נבנה משני מקורות, דרך **שני גנרטורים** שכותבים לאותה סכימה:

| גנרטור | מיקום ב־SeforimLibrary | קלט | מה מייצר |
|---|---|---|---|
| **sefariasqlite** | `generator/sefariasqlite/` | ייצוא Sefaria (`merged.json`, סכימות, `links/*.csv`) | ספרי ספריא + **כל** ארבעת המנגנונים |
| **otzariasqlite** | `generator/otzariasqlite/` | ספרי ה־repo הזה (`.txt`/`.docx` + `links/<ספר>_links.json` + `alt_toc/<ספר>_alt_toc.json`) | ספרי אוצריא + קישורים + עוגני־מילה + טווחים + עליות |

שרשרת הבנייה: קודם נבנה בסיס Sefaria, ואז `otzariasqlite` **מוסיף עליו** את ספרי אוצריא והקישורים שלהם (Phase 1 — שורות; Phase 2 — קישורים, נקודת הכניסה `generateLinksOnly()`). לכן ספר של אוצריא יכול לקשר גם לספר של ספריא ולהפך.

מקור כל ספר נשמר בטבלת `source` (למשל `Sefaria`, `Tashma`...). זה משמש להכרעות דילוג (ראו פרק 3).

> הערה על ענפים: הקוד הקנוני נמצא בענף **`otzaria`** של SeforimLibrary (ה־main של הפורק). כל השורות והתכונות שמתוארות כאן קיימות שם ובעץ העבודה הפעיל.

<a name="2-סכימת-הdb"></a>
## 2. סכימת ה־DB

הסכימה מוגדרת ב־`dao/src/commonMain/sqldelight/io/github/kdroidfilter/seforimlibrary/db/Database.sq`.

### הקישור עצמו

```sql
-- Database.sq:315
CREATE TABLE link (
    id INTEGER PRIMARY KEY NOT NULL,
    sourceBookId INTEGER NOT NULL,
    targetBookId INTEGER NOT NULL,
    sourceLineId INTEGER NOT NULL,      -- שורה אחת בצד המקור
    targetLineId INTEGER NOT NULL,      -- שורה אחת בצד היעד
    targetLineIndex INTEGER NOT NULL,
    targetBookOrderIndex INTEGER NOT NULL,   -- דה־נורמליזציה למיון מפרשים
    connectionTypeId INTEGER NOT NULL,
    isDeclaredBase INTEGER NOT NULL DEFAULT 0  -- 1 = ספריא הצהירה מפורשות שזה ספר־בסיס
);
CREATE TABLE connection_type ( id INTEGER PRIMARY KEY, name TEXT UNIQUE );
```

**נקודת מפתח:** לקישור יש **שורת־מקור יחידה ושורת־יעד יחידה**. אין ב־`link` שום עמודת "עד שורה" ושום עמודת "מאיזה תו". שני הצרכים האלה (טווח ומילה) מטופלים בטבלאות־לוויין נפרדות:

### עוגני־מילה, טווחים, כיסוי

```sql
-- Database.sq:367 — קישורים למילה
CREATE TABLE link_anchor (
    linkId INTEGER NOT NULL,
    side INTEGER NOT NULL DEFAULT 0,    -- 0 = צד המקור, 1 = צד היעד
    charStart INTEGER NOT NULL,         -- אופסט תו גלוי (HTML מקולף)
    charEnd INTEGER,                    -- NULL = עוגן־נקודה (סמן), אחרת קצה־טווח
    label TEXT,                         -- הסמן המודפס, למשל "א"
    PRIMARY KEY (linkId, side, charStart)
);

-- Database.sq:380 — קישורי טווח
CREATE TABLE link_range (
    linkId INTEGER NOT NULL,
    side INTEGER NOT NULL,
    endLineId INTEGER NOT NULL,         -- השורה האחרונה שהטווח מכסה
    endLineIndex INTEGER NOT NULL,      -- דה־נורמליזציה של lineIndex
    PRIMARY KEY (linkId, side)
);

-- Database.sq:397 — שורות מכוסות (אחרי הראשונה)
CREATE TABLE link_coverage (
    lineId INTEGER NOT NULL,
    linkId INTEGER NOT NULL,
    side INTEGER NOT NULL,
    PRIMARY KEY (lineId, linkId, side)
);
```

### מבנה הכותרות החלופי (עליות)

```sql
-- Database.sq:470 — למשל "Parasha" של בראשית
CREATE TABLE alt_toc_structure (
    id INTEGER PRIMARY KEY NOT NULL,
    bookId INTEGER NOT NULL,
    key TEXT NOT NULL,                  -- מפתח יציב, למשל "Parasha"
    title TEXT, heTitle TEXT,
    UNIQUE (bookId, key)
);
-- Database.sq:484 — כל פרשה, וכל עלייה תחתיה
CREATE TABLE alt_toc_entry (
    id INTEGER PRIMARY KEY NOT NULL,
    structureId INTEGER NOT NULL,
    parentId INTEGER,                   -- עלייה מצביעה על הפרשה שמעליה
    textId INTEGER NOT NULL,            -- → tocText (טקסט מנוכרז)
    level INTEGER NOT NULL,             -- 0 = פרשה, 1 = עלייה
    lineId INTEGER,                     -- השורה שבה הפרשה/עלייה מתחילה
    isLastChild INTEGER NOT NULL DEFAULT 0,
    hasChildren INTEGER NOT NULL DEFAULT 0
);
-- Database.sq:505 — מיפוי כל שורה לרשומת־העלייה שמעליה
CREATE TABLE line_alt_toc (
    lineId INTEGER NOT NULL,
    structureId INTEGER NOT NULL,
    altTocEntryId INTEGER NOT NULL,
    PRIMARY KEY (lineId, structureId)
);
```

ה־TOC ה"רגיל" (`tocEntry`/`tocText`/`line_toc`) הוא בעל אותו מבנה, אבל תלוי־ספר במקום תלוי־מבנה, ושני הגנרטורים בונים אותו.

<a name="3-קישורים-רגילים"></a>
## 3. קישורים רגילים

### מה זה
קישור בין שורה אחת בספר־מקור לשורה אחת בספר־יעד, עם **סוג קשר** (`connection_type`). זה מה שמזין את פאנל המפרשים/מקורות באפליקציה.

`connection_type` — 14 ערכים (מסודרים כפי ה־enum `ConnectionType` ב־`core/.../models/Link.kt:57`):

```
1 COMMENTARY   2 SUPER_COMMENTARY   3 TARGUM   4 REFERENCE
5 SOURCE*      6 MIDRASH   7 QUOTATION   8 MESORAT_HASHAS
9 EIN_MISHPAT  10 DIBUR_HAMATCHIL  11 PARSHANUT
12 MISHNAH_IN_TALMUD  13 RELATED  14 OTHER
```
`SOURCE` הוא **וירטואלי — לעולם לא נשמר**. כשמפרש X מקושר לבסיס Y כ־COMMENTARY, הכיוון ההפוך ("Y הוא מקור של X") נגזר בזמן קריאה ע"י היפוך הקישור בשאילתות־המראה. אל תסתמכו על מספר ה־id — הצטרפו לפי `connection_type.name`.

### יצירה במסלול Sefaria
קלט: `links/*.csv` עם העמודות `Citation 1`, `Citation 2`, `Conection Type` (שגיאת־כתיב מקורית של ספריא, במכוון). הזרימה (`generator/sefariasqlite/.../SefariaLinksImporter.kt`, `processLinkFile`):

1. **פתרון ref→שורה** (`SefariaImportRefs.kt`, `resolveRefs`): כל ציטוט מנורמל (`canonicalCitation`) ומתאים מול מפת השורות של הספר.
2. **בחירת סוג** (`ConnectionType.fromString`): מיפוי מחרוזת ה־CSV. ריק/`none`/לא־מוכר → `OTHER`. סוג ריק מנסים לשחזר מהסכימה (`inferConnectionTypeFromSchema`).
3. **אוריינטציה + `isDeclaredBase`** (`resolveDirectionalConnectionTypesForMeta`): מי הבסיס ומי התלוי נקבע לפי עדיפות יורדת — הצהרת `base_text_titles` מפורשת → אסימטריית `dependence` → דגל `isBaseBook` → `priorityRank` → אחרת הורדה ל־`OTHER`. `isDeclaredBase=1` **רק** כשספריא הצהירה מפורשות (משמש למיון בתצוגת "מקורות").
4. **id יציב + כתיבה**: `linkId` מוקצה דטרמיניסטית מ־`(srcLineId, tgtLineId, connectionTypeId)` (יציב בין builds — קריטי ל־delta), ונכתב דרך `insertLinksBatch` (`SeforimRepository.kt`).

### יצירה במסלול אוצריא
קלט: קובץ JSON אחד לכל ספר־מקור, בשם `<כותרת>_links.json`, בתיקיית `links/` שלצד תיקיית `ספרים`. **מערך** של רשומות:

```json
{
  "line_index_1": 39,
  "heRef_2": "הערות",
  "path_2": "הערות על צלח על פסחים.txt",
  "line_index_2": 1,
  "Conection Type": "commentary"
}
```

- `line_index_1` — מספר השורה בספר־המקור (**1־based**; מומר ל־0־based בגנרטור).
- `path_2` — נתיב הקובץ של ספר־היעד; הכותרת נגזרת מהרכיב האחרון (בלי סיומת).
- `line_index_2` — מספר השורה בספר־היעד.
- `Conection Type` — עובר דרך `ConnectionType.fromString`.

הזרימה (`generator/otzariasqlite/.../Generator.kt`, `processLinksForBook`):
1. ספר־המקור = שם הקובץ בלי `_links`; ספר־היעד מ־`path_2` — שניהם ב־cache בזיכרון.
2. **דילוג Sefaria (per-link):** אם *שני* הצדדים הם ספרי ספריא — דילוג (כי ספריא כבר מספקת את הקישור). אם רק צד אחד ספריא (למשל משנה ברורה→שער הציון) — **נשמר**.
3. המרת שני האינדקסים ל־`lineId` (מטמון `lineIndex→lineId` בזיכרון).
4. **שער כותרות:** אם אחת השורות היא שורת־כותרת (`<h…>`) — הקישור מדולג.
5. כתיבת שורת `link` עם **id יציב** לפי `(sourceLineId, targetLineId, connectionTypeId)` — כמו ב־importer של ספריא: אותו זוג שורות עם סוג קשר שונה = שני קישורים נפרדים, וריצה חוזרת מקבלת את אותו id (delta). אם הרשומה נושאת עוגן/טווח — הם נכתבים בהמשך על אותו `linkId` (פרקים 4–5). קישורי־החברותא (`GenerateHavroutaLinks.kt`) מייצרים רק שורות `link`.

<a name="4-קישורים-למילה"></a>
## 4. קישורים למילה (עוגני־תו)

### מה זה
קישור רגיל (שורה↔שורה) שמעוגן בנוסף **למיקום־תו מדויק בתוך השורה** — כמו סמני האותיות (א), (ב), (ג) ב"שער הציון"/"משנה ברורה", או ציטוט מודגש בתוך פסקה. מיוצג ע"י רשומה ב־`link_anchor` שמצביעה על ה־`linkId`.

- `side` — 0 = העוגן על שורת־המקור, 1 = על שורת־היעד. (בבניית ספריא, מכיוון שהקישור מאוחסן תמיד בכיוון בסיס→תלוי, סמני־מפרש הם `side=0`.)
- `charStart`/`charEnd` — **אופסטי תו גלוי**: תגי HTML מקולפים, וכל ישות (`&amp;…;`) נספרת כתו אחד. `charEnd=NULL` → עוגן־נקודה (סמן בודד), אחרת מרווח מסומן.
- `label` — הסמן המודפס (למשל "א"). מרבית העוגנים ב־DB הם נקודתיים עם תווית קצרה (1–3 תווים).

כלל־ברזל בשני המסלולים: **עוגן נכתב רק אם כבר קיים `link` בין אותן שתי שורות.** אי אפשר "להמציא" קשר דרך עוגן.

### יצירה במסלול Sefaria (שני תת־מסלולים)
1. **`SefariaCharLevelAnchors`** — מתאי `Char Level Data` ב־CSV (מאתר ציטוטים). קלט JSON כמו `{"startChar":4639,"endChar":4657,"versionTitle":"…","language":"he"}`. שערי־דיוק קפדניים: מדלג על מבוסס־מילה, שפה≠עברית, ספר רב־גרסאי, אי־התאמת גרסה, אופסטים מחוץ לתחום — וכל דילוג נספר ומדווח, לעולם לא מנוחש. `label=null`.
2. **`SefariaInlineAnchors`** — מסמני `<i data-commentator="…" data-order="N" data-label="א">` שספריא מטמיעה בטקסט־הבסיס. מחלץ את האופסט הגלוי, בוחר את ספר־המפרש, מאמת מול קישור קיים, וכותב עוגן־נקודה `side=0` עם ה־`label`.

### יצירה במסלול אוצריא
זהו המסלול הרלוונטי ל"שער הציון". הפונקציה `buildLinkAnchor` (`Generator.kt:1370`) נקראת מיד אחרי כל `insertLink`:

```kotlin
val rawStart = linkData.start?.toInt() ?: return null   // אין start → אין עוגן
val content  = repository.getLine(sourceLineId)?.content // תוכן שורת־המקור
// בדיקת גבולות...
val rawEnd = linkData.end?.toInt()?.takeIf { it in rawStart..content.length }
return LinkAnchor(
    linkId, side = 0,                                    // תמיד צד־מקור
    charStart = countVisibleChars(content, rawStart),    // raw → תווים גלויים
    charEnd   = rawEnd?.let { countVisibleChars(content, it) },
    label     = anchorLabelFromHeRef(linkData.heRef_2),  // האות שאחרי " אות "
)
```

לכן, כדי לקבל עוגן־מילה, רשומת ה־`_links.json` חייבת לשאת שני שדות נוספים — `start` ו־`end` (אופסטי תו **גולמיים** לתוך שורת־המקור) — והתווית מחולצת מ־`heRef_2`:

```json
{
  "line_index_1": 2,
  "line_index_2": 2,
  "heRef_2": "שער הציון, סימן א, סעיף א אות ג",
  "path_2": "שער הציון.txt",
  "Conection Type": "linker",
  "start": 6,
  "end": 12
}
```

- `anchorLabelFromHeRef` מחפש `" אות "` ב־`heRef_2` ולוקח את מה שאחריו: `"…אות ג"` → `label="ג"`. בלי `" אות "` (למשל בקישורי־טווח של ה־linker) → `label=null`.
- `"Conection Type": "linker"` אינו ערך מוכר ולכן נשמר כ־`OTHER`, אבל **העוגן נכתב בכל זאת** (העוגן תלוי ב־`start`, לא בסוג).
- העוגן הוא **צד־מקור בלבד** (`side=0`). כדי שסמן־מילה יופיע בספר A המצביע ל־B, ספר A הוא ה־`line_index_1`/המקור, וה־`start`/`end` הם אופסטים בשורה של A.

**הדוגמה החיה (מתוך טסט הגנרטור `OtzariaLinkAnchorsTest.kt`):** מקור = "משנה ברורה" (ספריא), יעד = "שער הציון" (Tashma). מכיוון שרק צד אחד ספריא — הקישור נשמר. `start=6,end=12` בתוך `"אב <b>ג</b> דה"` מתכווצים ל־`charStart=3,charEnd=5` (התו `<b>ג</b>`), ו־`heRef_2="…אות ג"` נותן `label="ג"`. בדיוק הסמן שמופיע באפליקציה.

> מגבלה: פורמט ה־Map של DictaToOtzaria (`{שורה: [{start,end,refs}]}`) מזוהה אך **מודלג** (הממיר לא מומש) — נתוני האופסט שלו נזרקים כרגע.

<a name="5-קישורי-טווח"></a>
## 5. קישורי טווח

### מה זה
קישור שה־ref שלו מכסה **כמה שורות** — למשל `Exodus 1:1-6:1` (כל פרשת שמות) או `Rashi on Genesis 18:10:2-3`. המידול:
- שורת ה־`link` עצמה מעוגנת ב**שורה הראשונה** של הטווח (אפס שינוי במבנה הקישור).
- `link_range` שומר את השורה ה**אחרונה** (`endLineId`/`endLineIndex`).
- `link_coverage` — שורה אחת לכל שורה מכוסה **אחרי** הראשונה (שורות כותרת מוחרגות).

### יצירה במסלול Sefaria
- זיהוי: `CITATION_RANGE_REGEX` (`SefariaRangeRefs.kt`) — מקף בתוך ריצת־הכתובת המספרית בסוף הציטוט בלבד (מקף בתוך מילת־כותרת אינו טווח).
- פירוק: `parseCitationRange` — הקצה יורש רכיבים מובילים: `18:10:2-3` → קצה `18:10:3`; `6:1-6` → קצה `6:6`.
- פתרון הקצה: `PathRefPrefixIndex.lastUnder` ממפה תחילית־כתובת בכל עומק לשורה האחרונה תחתיה (כך שקצה שצוטט ברמת־ביניים, למשל `1`, נפתר לשורה האחרונה בפרק).
- כתיבה: `queueRangeSide` יוצר `LinkRange` + שורות `LinkCoverage`, ונכתב אחרי שכל שורות ה־`link` הוכנסו. `insertRange` הוא upsert ששומר את **הטווח הרחב ביותר** (דטרמיניסטי).
- תוקן באג "קילוף כפול" ב־`resolveRefs`: טווח שתחילתו היא־עצמה מפתח־בסיס (למשל `Ibn Ezra on Genesis 6:1-6`) היה נושר בשקט; התיקון מוסיף חיפוש verbatim לפני הקילוף השני.

### יצירה במסלול אוצריא: ✅ נתמך
הפורמט: שני שדות אופציונליים ברשומת `_links.json` — `line_index_1_end` (קצה הטווח בצד המקור) ו־`line_index_2_end` (בצד היעד), שניהם **1־based וכוללניים**, בדיוק כמו `line_index_X`:

```json
{ "line_index_1": 39,
  "line_index_2": 12, "line_index_2_end": 15,
  "heRef_2": "…", "path_2": "…", "Conection Type": "commentary" }
```

הסמנטיקה זהה לספריא (`queueRangeSide` ב־`Generator.kt`):
- הקישור עצמו נשאר מעוגן ב**שורת ההתחלה**; `link_range` מקבל את השורה האחרונה; `link_coverage` — כל שורה מכוסה אחרי הראשונה, **בדילוג שורות כותרת**.
- כל צד עצמאי — אפשר טווח רק במקור, רק ביעד, או בשניהם (`side` 0/1 בהתאם).
- טווח מנוון (`end == start`) = קישור רגיל, בלי רשומות טווח.
- קצה הפוך / מעבר לסוף הספר / שורת כותרת → **אזהרה בלוג והטווח מושמט** (הקישור הרגיל נשמר). דטרמיניסטי, בלי ניחושים.
- **הייבוא אוטוריטטיבי לקישורים המעובדים:** לפני כתיבת הטווחים נמחקות רשומות `link_range`/`link_coverage` ישנות של כל קישור שעובד בריצה — כך שקיצור או הסרה של `line_index_X_end` נקלטים בריצה חוזרת ולא משאירים שאריות.

> הערה: פייפליין ה־Python (`linker/to_otzaria_links.py`) עדיין מקפל טווחים לשורה בודדת (`min_link`) בעת הפקת ה־JSON — כלומר קבצים שהוא מפיק כיום אינם נושאים `line_index_X_end`. הפורמט והגנרטור כבר תומכים; עדכון ה־linker להפקת השדות הוא צעד המשך נפרד.

<a name="6-קישורי-כותרות--עליות"></a>
## 6. קישורי "כותרות" / עליות (מבנה חלופי)

### מה זה — וההבהרה
ב"בראשית", לצד ה־TOC הרגיל (פרק א'…פרק נ'), יש מבנה שני שמראה **פרשות** ותחת כל פרשה **עליות**: "עליה א", "עליה ב"… זהו **מבנה תוכן חלופי (alternate TOC)**, לא קישור. הוא נשמר ב־`alt_toc_*` ואין לו שום קשר לטבלת `link`.

מבנה אמיתי מה־DB (בראשית, `structureId=1`, `key="Parasha"`):
```
level 0  "בראשית"   (פרשה)   lineId → תחילת הפרשה
  level 1  "עליה א"           lineId → תחילת עלייה א'
  level 1  "עליה ב"           …
  …
level 0  "נח"       (פרשה)
  level 1  "עליה א"  …
```

### יצירה במסלול Sefaria
מקור: מפתח `alts` (או `alt_structs`) בסכימת הספר. כל פרשה היא node עם:
- `wholeRef` — טווח כל־הפרשה (למשל `"Genesis 1:1-6:8"`).
- `refs` — **מערך ref־ההתחלה של כל עלייה** (`"Genesis 1:1"`, `"Genesis 2:4"`, …).
- `heSectionNames` — למשל "עליה".

הבנייה (`generator/sefariasqlite/.../SefariaAltTocBuilder.kt`, `addEntry`):
1. יוצר `alt_toc_structure` עם `key`/`title`/`heTitle` (id יציב לפי `bookId+key`).
2. הפרשה נכנסת ב־`level 0`, טקסט = `heTitle`, ו־`lineId` נפתר מ־`wholeRef`/`refs` דרך `resolveLineForCitation`.
3. כל ref במערך `refs` נכנס כ־**ילד ב־`level 1`** עם `parentId`=הפרשה, ותווית שנבנית ב־`buildChildLabel`: בסיס "עליה" + גימטריה של האינדקס → "עליה א", "עליה ב"…
4. `line_alt_toc` ממפה **כל שורה** בספר לרשומת־העלייה/פרשה האחרונה שקדמה לה (הכותרת העמוקה ביותר בשורה מנצחת).
5. `hasChildren`/`isLastChild` מסומנים בסיום.

ה־TOC ה**רגיל** (`tocEntry`) נבנה מעץ ה־`schema` (level 0 = כותרת הספר, level 1 = "פרק א"…), ו־`line_toc` ממפה כל שורה לכותרת שקדמה לה — אותו רעיון מיפוי.

### יצירה במסלול אוצריא: ✅ נתמך
קלט: קובץ JSON לכל ספר בתיקיית `alt_toc/` (לצד `links/`), בשם `<כותרת ספר>_alt_toc.json` — המקבילה של `alts` בספריא, אבל עם **מספרי שורות** במקום refs ועם תוויות מפורשות במקום גימטריה אוטומטית:

```json
[
  { "key": "Parasha", "heTitle": "פרשה",
    "nodes": [
      { "heTitle": "בראשית", "line": 1, "children": [
          { "heTitle": "עליה א", "line": 1 },
          { "heTitle": "עליה ב", "line": 36 }
      ]},
      { "heTitle": "נח", "line": 154, "children": [ … ] }
    ]}
]
```

- הקובץ הוא **מערך מבנים** (ספר יכול לשאת כמה מבנים חלופיים, כמו בספריא). כל מבנה: `key` (חובה — מזהה יציב באנגלית, משמש ל־id delta־יציב), `title`/`heTitle` אופציונליים.
- כל node: `heTitle` (חובה — טקסט הכותרת כפי שיוצג), `line` (חובה — השורה שבה הוא מתחיל, **1־based** כמו `line_index_1`), `children` אופציונלי ורקורסיבי (level = עומק הקינון).
- הגנרטור (`processAltTocStructures` ב־`Generator.kt`) בונה `alt_toc_structure` + `alt_toc_entry` (כולל `hasChildren`/`isLastChild`), ממפה **כל שורה** בספר ב־`line_alt_toc` לכותרת הקרובה שקדמה לה (העמוקה מנצחת בשורה משותפת — כמו בספריא), ומדליק `book.hasAltStructures`.
- ולידציה: `line` מחוץ לטווח הספר או שורה כפולה תחת אותו הורה → **אזהרה והצומת מושמט** (ילדיו לא מעובדים). מבנה נבנה מחדש בשלמותו בכל ריצה (idempotent).
- **הקובץ אוטוריטטיבי לספרו:** `key` שהוסר מהמערך — המבנה שלו נמחק; `nodes` ריק — המבנה מוסר; **מחיקת הקובץ כולו** — כל מבני הספר נמחקים בריצה הבאה; עריכת `title`/`heTitle` מתעדכנת (בנייה-מחדש מלאה של שורת המבנה); `book.hasAltStructures` מתעדכן לשני הכיוונים. קובץ שמכוון לספר **ספריא** מדולג עם אזהרה, ומבני ספריא לעולם אינם נמחקים (הם בבעלות הסכימות של ספריא).

ה־TOC ה**רגיל** של ספר אוצריא ממשיך להיבנות מתגי `<h1>`–`<h6>` בטקסט; קובץ ה־alt_toc מוסיף מבנה **מקביל** (כמו פרשות/עליות לצד פרקים) בלי לגעת בו.

<a name="7-מטריצת-התמיכה"></a>
## 7. מטריצת התמיכה של הגנרטור של אוצריא

התשובה המרוכזת לשאלה "האם הגנרטור של אוצריא תומך בכל אלה" — **כן, בכל הארבעה**:

| מנגנון | טבלאות | אוצריא (`otzariasqlite`) | פרטים |
|---|---|---|---|
| **קישורים רגילים** | `link` | ✅ **נתמך** | `processLinksForBook` + חברותא; שורה↔שורה מ־`_links.json` |
| **קישורים למילה** | `link_anchor` | ✅ **נתמך** (חלקית) | `buildLinkAnchor`; **צד־מקור בלבד** (`side=0`), רק כשה־JSON נושא `start`, תווית מ־`" אות X"` ב־`heRef_2` |
| **קישורי טווח** | `link_range`/`link_coverage` | ✅ **נתמך** | `queueRangeSide`; שדות `line_index_1_end`/`line_index_2_end` ב־`_links.json`; טסט `OtzariaRangedLinksTest` |
| **כותרות / עליות** | `alt_toc_*` | ✅ **נתמך** | `processAltTocStructures`; קובצי `alt_toc/<ספר>_alt_toc.json`; טסט `OtzariaAltTocTest` |

הגבלות שנותרו: עוגני־מילה הם **צד־מקור בלבד** ורק כשקובץ הקישורים כולל `start`/`end`; פורמט ה־Map של DictaToOtzaria עדיין מודלג; ופייפליין ה־linker ה־Python טרם מפיק `line_index_X_end` (ראו פרק 5).

<a name="8-איך-ליצור-בפועל"></a>
## 8. איך ליצור בפועל (למתחזקי otzaria-library)

### קישור רגיל
הוסיפו רשומה ל־`links/<ספר-מקור>_links.json`:
```json
{ "line_index_1": 39, "line_index_2": 1,
  "heRef_2": "רש\"י", "path_2": "רש\"י על צלח על פסחים.txt",
  "Conection Type": "commentary" }
```
- האינדקסים **1־based** (שורה 1 = השורה הראשונה בקובץ).
- `path_2` — הנתיב היחסי לקובץ היעד; חייב להתאים לספר קיים.
- `Conection Type` נפוצים: `commentary`, `targum`, `reference`, `midrash`, `quotation` (לא־מוכר → `OTHER`).
- ודאו ששני הקצוות אינם שורות־כותרת (`<h…>`) — אחרת הקישור מדולג.

### קישור־למילה
כמו קישור רגיל, בתוספת `start`/`end` (**אופסטי תו גולמיים לתוך שורת־המקור**, כולל תגי HTML) ותווית ב־`heRef_2`:
```json
{ "line_index_1": 2, "line_index_2": 2,
  "heRef_2": "שער הציון, סימן א, סעיף א אות ג",
  "path_2": "שער הציון.txt",
  "Conection Type": "linker", "start": 6, "end": 12 }
```
- העוגן מופיע על **שורת־המקור** (`line_index_1`). לתווית, כללו `" אות X"` ב־`heRef_2` (X = אותיות עבריות, עד 6 תווים, אפשר עם גרשיים).
- `end` אופציונלי; בלעדיו → עוגן־נקודה.
- הדרך המומלצת אינה לכתוב ידנית אלא להריץ את פייפליין ה־**linker** (תיקיית `linker/`): הוא סורק את הטקסט, מזהה הפניות אוטומטית, ומייצר רשומות `"Conection Type": "linker"` עם `start`/`end` דרך `to_otzaria_links.py`.

### קישור־טווח
כמו קישור רגיל, בתוספת `line_index_1_end` ו/או `line_index_2_end` (1־based, כוללני):
```json
{ "line_index_1": 39,
  "line_index_2": 12, "line_index_2_end": 15,
  "heRef_2": "גמרא דף ג-ה", "path_2": "גמרא.txt", "Conection Type": "commentary" }
```
- הטווח יכול להיות בכל צד בנפרד או בשניהם; הקישור מעוגן תמיד בשורת ההתחלה.
- ודאו ש־end ≥ start ובתוך גבולות הספר, ושאינו שורת כותרת — אחרת הטווח יושמט עם אזהרה (הקישור יישמר כרגיל).
- שורות כותרת בתוך הטווח מדולגות אוטומטית מהכיסוי — אין צורך לפצל את הטווח סביבן.

### עליות / מבנה כותרות חלופי
צרו קובץ `alt_toc/<כותרת ספר>_alt_toc.json` (התיקייה לצד `links/`):
```json
[
  { "key": "Parasha", "heTitle": "פרשה",
    "nodes": [
      { "heTitle": "בראשית", "line": 1, "children": [
          { "heTitle": "עליה א", "line": 1 },
          { "heTitle": "עליה ב", "line": 36 }
      ]}
    ]}
]
```
- `key` באנגלית ויציב (אל תשנו אותו בין גרסאות — הוא בסיס ה־id ב־delta); `heTitle` הוא שם התצוגה.
- `line` הוא מספר השורה (1־based) שבו הכותרת מתחילה — מותר גם שורת כותרת (`<h…>`) וגם שורת תוכן, ומותר לפרשה ולעלייה הראשונה שלה לחלוק שורה (העלייה תנצח במיפוי).
- קינון חופשי דרך `children`; אין הגבלת עומק.
- זה מבנה **מקביל** ל־TOC הרגיל (תגי `<h>`), לא תחליף לו.

<a name="נספח-א-הapi-לכתיבה"></a>
## נספח א': ה־API לכתיבה ב־DB (למפתחי הגנרטור)

כל הכתיבה עוברת דרך `dao/.../SeforimRepository.kt`, ברוב הטבלאות עם id יציב מ־`InMemoryIdAllocator` (מפתח טבעי → id קבוע בין builds, לתמיכת delta). **חריג מכוון:** `alt_toc_entry` מוכנס ב־auto-id בשני הגנרטורים — קישור מפתח־טבעי לרשומות אלה נדחה ל־Phase 1.5 (ראו ההערה ב־`SefariaAltTocBuilder`), והמבנים נבנים-מחדש בשלמותם פר ספר, כך שה-ids הלא-יציבים עולים רק ב־DELETE+INSERT בפאץ' ולא בנכונות. מתודות מרכזיות:

| רשומה | מתודת repository | הערה |
|---|---|---|
| `link` | `bindings.insertLinkStable` (ספריא ואוצריא; id מ־allocator לפי src/tgt/type) · `insertLink`/`insertLinksBatch` (מסלולים ישנים/חברותא) | `targetBookOrderIndex` מחושב בזמן הכתיבה |
| `link_anchor` | `insertLinkAnchorsBatch` | `INSERT OR IGNORE` |
| `link_range` | `insertLinkRangesBatch` | upsert — הטווח הרחב מנצח |
| `link_coverage` | `insertLinkCoverageBatch` | `INSERT OR IGNORE` |
| `tocEntry` / `tocText` / `line_toc` | `insertTocEntry` / `getOrCreateTocText` / `insertLineTocBatch` | `tocText` מנוכרז לפי טקסט |
| `alt_toc_structure` / `alt_toc_entry` / `line_alt_toc` | `upsertAltTocStructure` / `insertAltTocEntry` / `upsertLineAltToc` | |
| `connection_type` | `insertConnectionTypeWithId` / `getOrCreateConnectionType` | 14 הערכים נזרעים מראש מ־`ConnectionType.values()` |
| `book_has_links` | `updateBookHasLinks` (SQL מרוכז) | + דגלי `book.hasXxxConnection` |

כל הטבלאות האלה רשומות ב־`PatchTables` (delta) וב־`LogicalContentHasher`. רובן `updatable`; `link_coverage` הוא junction טהור (`updatable=false`).
